# ユースケース記述（ホテル予約システム）

最新版。実装（`hotel-reservation-system`）の各ユースケースに対応。

---

## UC1. 空室を検索する

| 項目 | 内容 |
|------|------|
| 主アクター | 利用者 |
| 目的 | 利用者が希望する部屋タイプの空室を確認する |
| 事前条件 | システムが稼働している |
| 事後条件 | 条件に合う空室一覧または警告が表示される。部屋や予約の状態は変更されない |
| 基本フロー | 1. 利用者が検索する部屋タイプを入力する<br>2. システムが指定された部屋タイプの空室を検索する<br>3. システムが、各空室の部屋番号、部屋タイプ、基本宿泊料を一覧表示する |
| 代替フロー | 1a. 部屋タイプが入力されなかった場合：システムは全部屋タイプを対象として空室を検索する<br>2a. 条件に合う空室が存在しない場合：システムは「条件に合う空室がありません」と警告を表示し、処理を終了する<br>2b. 登録されていない部屋タイプが入力された場合：システムは条件に合う空室が存在しないものとして警告を表示する |

## UC2. 部屋を予約する

| 項目 | 内容 |
|------|------|
| 主アクター | 利用者 |
| 目的 | 利用者が希望する部屋タイプ・宿泊予定日で部屋を予約し、予約番号を得る |
| 事前条件 | システムが稼働している |
| 事後条件 | 予約が「予約中」で作成され、予約番号が発行される。対象の部屋が1室確保される |
| 基本フロー | 1. 利用者が利用者名、部屋タイプ、宿泊予定日を入力する<br>2. システムが該当する部屋タイプの空室を1室確保する<br>3. システムが予約を「予約中」の状態で作成し、予約番号を発行する<br>4. システムが予約番号と部屋番号を表示する |
| 代替フロー | 1a. 利用者名が入力されなかった場合：システムは「利用者名が不正です」と表示し、処理を終了する<br>1b. 部屋タイプが入力されなかった場合：システムは「部屋タイプが不正です」と表示し、処理を終了する<br>1c. 宿泊予定日の形式が不正な場合：システムは「日付の形式が不正です」と表示し、処理を終了する<br>2a. 空室が存在しない場合：システムは「空室がありません」と警告を表示し、処理を終了する |

## UC3. チェックインする

| 項目 | 内容 |
|------|------|
| 主アクター | ホテル受付係（利用者が予約番号を提示する） |
| 目的 | 予約情報を確認し、利用者のチェックイン手続きを完了する |
| 事前条件 | システムが稼働し、ホテル受付係が利用者から予約番号を受け取っている |
| 事後条件 | 予約が「宿泊中」になり、宿泊記録が生成され、対象の部屋が使用中になる |
| 基本フロー | 1. ホテル受付係が予約番号を入力する<br>2. システムが予約を検索し、予約情報を表示する<br>3. システムが予約状態が「予約中」であることを確認する<br>4. ホテル受付係が受付係名を入力してチェックインを確定する<br>5. システムが予約を「宿泊中」に変更する<br>6. システムが対象の部屋を使用中にし、宿泊記録を生成する<br>7. システムがチェックイン完了メッセージと部屋番号を表示する |
| 代替フロー | 2a. 予約が存在しない場合：システムは「予約が見つかりません」と表示し、処理を終了する<br>3a. 予約がすでに「宿泊中」の場合：システムは「既にチェックイン済みの予約です」と表示し、処理を終了する<br>3b. 予約が「キャンセル」の場合：システムは「キャンセル済みの予約はチェックインできません」と表示し、処理を終了する<br>4a. 受付係名が入力されなかった場合：システムは受付係名を「-」として処理を続行する |

> **既知の不具合:** 現在の実装では、キャンセル済み予約に対するチェックインを拒否する処理が不足している。そのため、代替フロー3bを満たすには、予約状態が「予約中」の場合に限ってチェックインを許可するよう修正する必要がある。

## UC4. チェックアウトする

| 項目 | 内容 |
|------|------|
| 主アクター | ホテル受付係（利用者が部屋番号を提示する） |
| 目的 | 利用者の宿泊記録を確認し、支払いとチェックアウトの手続きを完了する |
| 事前条件 | システムが稼働し、ホテル受付係が利用者から部屋番号を受け取っている |
| 事後条件 | チェックアウト日時と支払い記録が生成され、対象の部屋が空室に戻る |
| 基本フロー | 1. ホテル受付係が部屋番号を入力する<br>2. システムが宿泊中の宿泊記録を検索し、宿泊情報を表示する<br>3. ホテル受付係が支払いを確定する<br>4. システムがチェックアウト日時を記録する<br>5. システムが部屋の基本宿泊料を支払額として支払い記録を生成する<br>6. システムが対象の部屋を空室に戻す<br>7. システムが支払額を表示する |
| 代替フロー | 1a. 部屋番号が整数でない場合：システムは「部屋番号が不正です」と表示し、処理を終了する<br>2a. 対象の部屋に宿泊中の宿泊記録が存在しない場合：システムは「宿泊データが見つかりません」と表示し、処理を終了する |

## UC5. 部屋を管理する

| 項目 | 内容 |
|------|------|
| 主アクター | 管理者 |
| 目的 | 管理者が新しい部屋を登録し、登録済みの客室情報と空室状態を確認する |
| 事前条件 | システムが稼働している |
| 事後条件 | 部屋を追加した場合、新しい部屋が空室状態で登録される。一覧表示の場合、登録情報は変更されない |
| 基本フロー | 1. 管理者が部屋管理機能を選択する<br>2. システムが「部屋を追加する」と「登録済みの全客室を表示する」の選択肢を表示する<br>3. 管理者が実行する処理を選択する |
| 代替フロー | 3a. 管理者が「部屋を追加する」を選択した場合：管理者が部屋番号、部屋タイプ、基本宿泊料を入力する。システムは新しい部屋を空室状態で登録し、登録完了メッセージを表示する<br>3b. 管理者が「登録済みの全客室を表示する」を選択した場合：システムは各客室の部屋番号、部屋タイプ、基本宿泊料、空室状態を一覧表示する<br>3c. 選択肢以外の値が入力された場合：システムは「1または2を入力してください」と表示し、処理を終了する<br>3a-1. 部屋番号または基本宿泊料が整数でない場合：システムは入力が不正であることを表示し、部屋を登録せずに処理を終了する<br>3a-2. 部屋タイプが入力されなかった場合：システムは「部屋タイプが不正です」と表示し、部屋を登録せずに処理を終了する<br>3a-3. 入力された部屋番号がすでに登録されている場合：システムは「部屋番号が既に存在します」と表示し、部屋を登録せずに処理を終了する |

## UC6. 予約をキャンセルする（保守工程で追加）

| 項目 | 内容 |
|------|------|
| 主アクター | 利用者 |
| 目的 | 利用者が予約済みの宿泊を取り消す |
| 事前条件 | システムが稼働し、利用者が予約番号を保持している |
| 事後条件 | 予約が「キャンセル」になり、確保されていた部屋が空室に戻る |
| 基本フロー | 1. 利用者が予約番号を入力する<br>2. システムが予約を検索し、予約情報を表示する<br>3. システムが予約状態が「予約中」であることを確認する<br>4. 利用者がキャンセルを確定する<br>5. システムが予約状態を「キャンセル」に変更する<br>6. システムが確保されていた部屋を空室に戻す<br>7. システムがキャンセル完了メッセージを表示する |
| 代替フロー | 2a. 予約が存在しない場合：システムは「予約が見つかりません」と表示し、処理を終了する<br>3a. 予約がすでに「宿泊中」の場合：システムは「チェックイン済みの予約はキャンセルできません」と表示し、処理を終了する<br>3b. 予約がすでに「キャンセル」の場合：システムは「既にキャンセル済みの予約です」と表示し、処理を終了する |
